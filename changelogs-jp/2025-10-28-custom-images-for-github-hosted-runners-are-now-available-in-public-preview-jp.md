# GitHub ホストランナー用のカスタムイメージがパブリックプレビューで利用可能に

## Overview
GitHub ホストランナー用のカスタムイメージがパブリックプレビューで利用可能になりました。この機能により、GitHub が提供するベースイメージから始めて、ワークフローのニーズに合わせた独自の仮想マシンイメージを構築できます。

## Detailed Explanation
### 概要
- GitHub ホストランナー用のカスタムイメージがパブリックプレビューで利用可能になりました。この機能により、GitHub が提供するベースイメージから始めて、ワークフローのニーズに合わせた独自の仮想マシンイメージを構築できます。
- カスタムイメージには、リポジトリコード、事前プルされたコンテナイメージ、必要なバイナリ、証明書、その他の依存関係を含めることができます。開発者は次のメリットを得られます：
- より高速なワークフロー：事前インストールされたツールと依存関係でジョブを開始し、セットアップ時間を短縮してビルドを大幅に高速化します。
- 強化されたセキュリティ：ビルド環境を標準化し、組織全体のイメージポリシーを実施し、攻撃面を減らすことで脆弱性を最小限に抑えます。
- コスト最適化：ジョブの起動が速くなることで課金される分数が減り、全体的な CI/CD コストを削減できます。
- エンタープライズグレードのカスタマイズ：チームが必要とするものを正確に事前インストールします。証明書やプロキシから監視エージェントまで、イメージのライフサイクルとロールアウトを完全に制御できます。
- ライフサイクル管理：管理者は、イメージごとのバージョン数、所定の期間使用されていないイメージの削除、組織のセキュリティとコンプライアンス要件に合わせて無効にされる前の最大イメージ期間のポリシーを設定できます。

### 開始方法
- カスタムイメージを使用すると、GitHub Actions ランナー環境の正確な構成を定義でき、すべてのワークフローは必要なツール、依存関係、設定がすでに配置された状態で開始されます。開始方法は次のとおりです：
- 1. カスタムイメージを生成できるより大きなランナーを作成します。カスタムイメージを構築できるランナーをセットアップすることから始めます。このランナーは、ベースイメージ（GitHub パートナーイメージまたはすでに生成したもの）を新しいイメージの基礎として使用します。この階層的なアプローチにより、信頼できる環境の上に構築し、必要なものだけを追加できます。
- 2. イメージ生成ワークフローを実行します。ワークフローファイルで、ジョブに新しい snapshot タイプを追加します。これにより、必要なすべてのツールとパッケージをインストールした後のランナーの状態を GitHub Actions がキャプチャするよう指示します。
- 3. カスタムイメージを使用してランナーをランナーグループに追加します。イメージがビルドされると、より大きなランナーがそれを使用し、ランナーグループに割り当てられます。指定されたイメージをすでに使用しているランナーは、イメージの新しいバージョンを自動的に取得します。ワークフローでは、ランナーまたはランナーグループ名を参照する必要があります—イメージ名だけでランナーをターゲットにすることはできません。このランナーを使用する以降のすべてのワークフロー実行は、カスタム環境で開始され、時間を節約し、一貫性を確保します。イメージが作成されてから新しいバージョンが使用可能になるまで、10〜15分かかります。
- 4. 今後のワークフローでカスタムイメージを使用します。カスタムイメージを使用するには、ワークフローの runs-on ターゲットをカスタムイメージを使用しているランナー（またはランナーを含むランナーグループ）の名前に設定します。すべてのジョブは依存関係が事前インストールされた状態で開始されるため、ビルドとシッピングに集中できます。
