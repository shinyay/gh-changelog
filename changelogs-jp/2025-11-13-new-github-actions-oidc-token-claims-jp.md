# GitHub ActionsのOIDCトークンクレームの追加

## Overview
GitHub ActionsのOpenID Connect（OIDC）トークンクレームに`check_run_id`が含まれるようになりました

## Detailed Explanation
### Overview
- GitHub ActionsのOpenID Connect（OIDC）トークンクレームに`check_run_id`が含まれるようになりました
- この機能強化により、きめ細かい属性ベースのアクセス制御が可能になり、外部サービスと統合するワークフローの監査性が向上します。プラットフォームチームは、コンプライアンスとトレーサビリティのために、OIDCトークンを特定のジョブとそれを生成したコンピュートリソースに関連付ける必要があることがよくあります。例えば、ワークフローがAzureでホストされている内部サービスを呼び出す場合、チームは監査のためにトークンをジョブ実行にリンクする必要があります。以前は、トークンに正確なジョブを特定する方法がありませんでした。既存のクレームである`run_id`と`run_attempt`と並んで`check_run_id`が追加されたことで、以下が可能になります：
- トークンを、リクエストを実行した正確なジョブとコンピュートリソースまでトレースできます。
- すべてのリポジトリを列挙することなく、最小権限ポリシーを実装できます。
- シークレット漏洩リスクを軽減し、失効を迅速化できます。
- アクセスをガバナンスされたリポジトリの状態にマッピングすることで、コンプライアンスと監査ワークフローを改善できます。
- OIDCとトークンの設定方法の詳細については、ドキュメントをご覧ください。

GitHub ActionsのOpenID Connect（OIDC）トークンクレームに`check_run_id`が含まれるようになりました


この機能強化により、きめ細かい属性ベースのアクセス制御が可能になり、外部サービスと統合するワークフローの監査性が向上します。プラットフォームチームは、コンプライアンスとトレーサビリティのために、OIDCトークンを特定のジョブとそれを生成したコンピュートリソースに関連付ける必要があることがよくあります。例えば、ワークフローがAzureでホストされている内部サービスを呼び出す場合、チームは監査のためにトークンをジョブ実行にリンクする必要があります。以前は、トークンに正確なジョブを特定する方法がありませんでした。既存のクレームである`run_id`と`run_attempt`と並んで`check_run_id`が追加されたことで、以下が可能になります：


* トークンを、リクエストを実行した正確なジョブとコンピュートリソースまでトレースできます。
* すべてのリポジトリを列挙することなく、最小権限ポリシーを実装できます。
* シークレット漏洩リスクを軽減し、失効を迅速化できます。
* アクセスをガバナンスされたリポジトリの状態にマッピングすることで、コンプライアンスと監査ワークフローを改善できます。


OIDCとトークンの設定方法の詳細については、[ドキュメント](https://docs.github.com/actions/concepts/security/openid-connect)をご覧ください。
